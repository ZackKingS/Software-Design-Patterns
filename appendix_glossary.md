# 附录：模式速查索引 & 术语表

> 读完全书后，这一页用来"快速定位"：忘了某个模式讲在哪、解决什么问题，或某个术语什么意思，回这里查。

## A. 设计原则速查（SOLID）

| 原则 | 一句话 | 章节 |
| --- | --- | --- |
| **S** 单一职责 (SRP) | 一个类只负责一件事，只有一个变化的理由 | [第1章](chapter_1.md) |
| **O** 开闭原则 (OCP) | 对扩展开放、对修改关闭——加功能靠新增，别改老代码 | [第1章](chapter_1.md) |
| **L** 里氏替换 (LSP) | 子类必须能无差别替换父类，别破坏父类的约定 | [第1章](chapter_1.md) |
| **I** 接口隔离 (ISP) | 别强迫一个类依赖它用不到的方法，接口要小而专 | [第1章](chapter_1.md) |
| **D** 依赖倒置 (DIP) | 面向接口编程，高层和低层都依赖抽象 | [第1章](chapter_1.md) |

## B. 创建型模式（怎么"造"对象）

| 模式 | 解决什么 | 一句话定位 | 章节 |
| --- | --- | --- | --- |
| **单例 Singleton** | 全局唯一实例 | 保证一个类只有一个对象，并提供全局访问点 | [第3章](chapter_3.md) |
| **工厂 Factory** | 创建逻辑散落、`new` 满天飞 | 把"造哪个对象"的逻辑封装进工厂，调用方只面向接口 | [第4章](chapter_4.md) |
| **建造者 Builder** | 参数又多又杂的伸缩构造器 | 链式分步构建复杂对象，最后 `build()` 收工 | [第5章](chapter_5.md) |
| **原型 Prototype** | 创建成本高、需大量相似对象 | 拿现成对象当模板 `clone()` 复制（注意深浅拷贝） | [第6章](chapter_6.md) |

## C. 结构型模式（怎么"拼"对象）

| 模式 | 解决什么 | 一句话定位 | 章节 |
| --- | --- | --- | --- |
| **适配器 Adapter** | 两个接口不兼容 | 加个"转接头"做接口转换，让它们能协作 | [第7章](chapter_7.md) |
| **装饰器 Decorator** | 功能组合导致类爆炸 | 层层包裹动态增强功能，不改接口 | [第8章](chapter_8.md) |
| **代理 Proxy** | 核心逻辑被权限/日志等杂活污染 | 配个"替身"控制访问、在前后把关 | [第9章](chapter_9.md) |
| **外观 Facade** | 调用方要操作一堆子系统 | 提供一个统一的简单入口，屏蔽内部复杂度 | [第10章](chapter_10.md) |
| **桥接 Bridge** | 两个独立维度用继承导致"相乘"爆炸 | 把维度拆开用组合连接，"相乘"变"相加" | [第11章](chapter_11.md) |
| **组合 Composite** | 树形结构里到处 `instanceof` | 让叶子和容器实现同一接口，统一递归处理 | [第12章](chapter_12.md) |
| **享元 Flyweight** | 大量重复对象撑爆内存 | 分离内/外部状态，共享只读的内部状态 | [第13章](chapter_13.md) |

## D. 行为型模式（对象之间怎么"协作"）

| 模式 | 解决什么 | 一句话定位 | 章节 |
| --- | --- | --- | --- |
| **策略 Strategy** | 一个方法里塞满算法 if-else | 把每种算法封装成可替换的策略对象 | [第14章](chapter_14.md) |
| **观察者 Observer** | 一个对象变了要通知一大群 | 发布者维护订阅名单，变化时群发通知 | [第15章](chapter_15.md) |
| **命令 Command** | 请求发起者和执行者绑死 | 把"请求"封装成对象，可传递/排队/撤销 | [第16章](chapter_16.md) |
| **模板方法 Template Method** | 流程骨架被反复复制 | 父类定流程骨架，子类只填变化的"空格" | [第17章](chapter_17.md) |
| **状态 State** | 每个方法里都塞满状态 if-else | 把每个状态封装成类，由状态自己决定行为 | [第18章](chapter_18.md) |
| **责任链 Chain of Responsibility** | 分级处理的 if-else | 把处理者串成链，请求沿链"击鼓传花"传递 | [第19章](chapter_19.md) |

| **中介者 Mediator** | 对象之间互相引用成网状 | 用中介对象收口交互，"网状"变"星形" | [第22章](chapter_22.md) |
| **访问者 Visitor** | 结构稳定但操作频繁增加 | 把操作抽成访问者，加操作不动元素类 | [第23章](chapter_23.md) |
| **解释器 Interpreter** | 自定义小语言/规则求值 | 每条语法规则一个类，解析成树递归求值 | [第24章](chapter_24.md) |

## E. 术语表

按拼音/英文首字母排序，方便检索。

- **抽象语法树 (AST)**：把一段代码/规则解析成的树形结构，每个节点是一个语法元素。访问者、解释器的主场。见 [第23章](chapter_23.md)、[第24章](chapter_24.md)。
- **对象适配器 / 类适配器**：适配器的两种实现。对象适配器靠**组合**持有被适配者（推荐），类适配器靠**继承**。见 [第7章](chapter_7.md)。
- **DCL（双重检查锁，Double-Checked Locking）**：懒汉式单例的线程安全写法，两层 `if` + `synchronized` + `volatile`。见 [第3章](chapter_3.md)。
- **fail-fast**：集合在遍历中被结构性修改时立即抛 `ConcurrentModificationException` 的保护机制。见 [第20章](chapter_20.md)。
- **饿汉式 / 懒汉式**：单例的两种创建时机——类加载即创建 / 用到才创建。见 [第3章](chapter_3.md)。
- **good code / bad code（✅/❌）**：本书代码对照惯例，❌ 是被批判的反面写法，✅ 是重构后的正解。
- **钩子方法 (Hook)**：模板方法里父类提供默认实现、子类可选择性覆盖的方法，用来微调流程。见 [第17章](chapter_17.md)。
- **好莱坞原则 (Hollywood Principle)**："别打电话给我们，我们会打给你"——父类/框架控制流程，反过来回调你的代码。见 [第17章](chapter_17.md)。
- **内部状态 / 外部状态 (Intrinsic / Extrinsic)**：享元模式对对象状态的拆分——可共享的（内部）与每次传入的（外部）。见 [第13章](chapter_13.md)。
- **类爆炸**：用继承/穷举组合表达多维变化时，类的数量指数级膨胀。装饰器、桥接、享元都在治它。
- **双分派 (Double Dispatch)**：访问者模式靠 `accept` + `visit` 两次回调，同时根据"元素类型"和"操作类型"确定执行逻辑。见 [第23章](chapter_23.md)。
- **深拷贝 / 浅拷贝**：复制对象时是否连引用指向的内容也复制。浅拷贝共享引用（易踩坑），深拷贝彻底独立。见 [第6章](chapter_6.md)、[第21章](chapter_21.md)。
- **上帝类 (God Class)**：什么都管、无所不知的巨型类。外观和中介者用不好就会退化成它。见 [第10章](chapter_10.md)、[第22章](chapter_22.md)。
- **伸缩构造器 (Telescoping Constructor)**：参数又多又杂、还得一堆重载的构造方法反模式，建造者模式专治。见 [第5章](chapter_5.md)。
- **动态代理 / 静态代理**：静态代理手写代理类；动态代理运行时自动生成，是 Spring AOP 底层。见 [第9章](chapter_9.md)。
- **迪米特法则 (Law of Demeter / 最少知识原则)**：一个对象应尽量少地了解其他对象，只和"直接朋友"打交道。外观模式的理论基础。见 [第10章](chapter_10.md)。
- **反模式 (Anti-Pattern)**：常见但有害的写法。如伸缩构造器；单例用不好也常被视为反模式。
- **依赖注入 (DI)**：把依赖从外部"塞"进来（构造器/setter），而非内部自己 `new`。依赖倒置的落地手段。见 [第1章](chapter_1.md)、[第14章](chapter_14.md)。
